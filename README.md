Un LLM est structurellement non fiable. Il l'est indépendamment de son fournisseur, de son hébergement, de ses paramètres de sécurité. Cette non-fiabilité n'est pas un défaut corrigible, c'est une propriété de l'objet.

SSCG part de ce constat et ne cherche pas à le contourner. Il le traite comme une contrainte fixe, puis pose une question différente : comment utiliser un modèle non fiable pour administrer un système qui compte, sans que sa non-fiabilité représente une surface d'exposition pour ce système.

La réponse n'est pas cryptographique. Elle est structurelle. Le système réel n'est jamais projeté dans l'espace du LLM. Il est traduit dans un vocabulaire symbolique fermé, un ensemble de rôles abstraits, typés, sans ancrage au réel. Le LLM opère sur ces symboles avec une compétence complète. Il ne sait pas ce qu'ils désignent. Cette ignorance n'est pas une limitation, c'est la propriété qui rend le reste possible.

La séparation entre le monde symbolique et le monde réel est maintenue par un moteur local qui détient seul la table de résolution. Ce moteur ne raisonne pas. Il traduit, vérifie, exécute. Le LLM ne le perçoit pas. Il ne perçoit pas non plus la frontière qu'il ne franchit jamais.
